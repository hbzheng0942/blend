# Spike S1 结果 — 操作符 × 模型保真度矩阵

> Phase 0 产出 · 2026-07-13 · 执行：Claude Code
> 可视对比页：[spike-matrix.html](spike-matrix.html)（30 格主矩阵 + inject v2 复测 + creative 加测 + 多图探测，直接浏览器打开）
> 原始数据：`spike/results.json`、`spike/probe-results.json`、`spike/calls.jsonl`（全部 77+ 次调用逐条日志）

## TL;DR

| | 结论 |
|---|---|
| 可行性 | ✅ Agnes 免费档足以支撑 MVP 核心玩法，**建议进入 Phase 1** |
| 能打的操作符 | ⊕ fuse、⊃ absorb 开箱即用；→ inject 换强化 prompt 后达标 |
| 不能打的操作符 | ⊖ subtract、∩ intersect 两模型全灭（符合 PRD 风险表预判），MVP 置灰 |
| 多图上限 | **两模型均为 6 张**（服务端 400 明示 "at most 6 allowed"），级联执行器阈值取 6 |
| Data URI | ✅ 完全可用，1024px PNG（约 1.7MB/张）× 2 张请求体无压力 |
| 限流真相 | 免费档**从未返回 429**；压力信号是 **503 `image queue is full`** + 连接层异常，"熔炉过热"机制须按此适配 |

## 1. 测试设置

- 模型：`agnes-image-2.0-flash`（size=1024x1024）、`agnes-image-2.1-flash`（size=1K, ratio=1:1）
- 3 组测试输入（全部由 2.0 t2i 生成，1024×1024，见 `spike/inputs/`）：
  - **组A 物体+物体**：陶瓷茶壶 + 老式打字机
  - **组B 物体+材质**：白色高帮帆布鞋 + 熔岩纹理
  - **组C 物体+生物**：红色电话亭 + 橙色章鱼
- 输入一律 Data URI base64 经 `extra_body.image[]` 传入，prompt 使用 PRD 1.2 原版骨架
- 判定标准（PRD）：语义意图肉眼一致 ≥ 2/3 组则 pass

## 2. 操作符判定（30 格主矩阵）

| 操作符 | 2.0-flash | 2.1-flash | 逐组观察 |
|---|---|---|---|
| ⊕ fuse | ✅ PASS (2.5/3) | ✅ PASS (2.5/3) | A 组两模型都产出打字机×茶壶×龙形的惊艳融合物（2.1 更彻底）；B 组 2.0 完美（焦黑熔岩鞋）、2.1 过度发挥成熔岩龙（丢了鞋的特征）；C 组偏"同框缠绕"而非融为一体，勉强及格 |
| → inject（原版 prompt） | ❌ FAIL (0/3) | ❌ FAIL (0/3) | 系统性失效：模型把 image 2 理解为**场景/背景**而非材质源——B 组产出"白鞋摆在熔岩上"而非"熔岩材质的鞋"，A 组直接返回打字机原图 |
| → inject（v2 强化 prompt） | ✅ PASS | ✅ PASS | 见 §3，改写后 B 组两模型都完美命中 |
| ⊖ subtract | ❌ FAIL (0/3) | ❌ FAIL (0/3) | 退化为"返回 image 2 原图"或"两物同框"，从未发生特征剥离 |
| ∩ intersect | ❌ FAIL (0/3) | ❌ FAIL (0/3) | 同样退化；C 组甚至只返回"海底的章鱼"（电话亭完全丢失） |
| ⊃ absorb | ⚠️ 勉强 (1.5/3) | ✅ PASS (2.5/3) | B 组两模型都对（鞋面嵌入熔岩碎屑）；2.1 的 C 组把海洋元素碎片嵌进电话亭窗格，效果最好；A 组普遍退化 |

**核心发现**：两模型对双图输入有强烈的「主体 + 场景」先验。要求**产出单一新物体**的操作符（fuse/absorb）能对抗这个先验；要求**抽象推理**的操作符（subtract/intersect）完全无法用 prompt 骨架驱动。

## 3. inject v2 强化 prompt 复测

原版骨架失效后，用更强措辞复测（明确"物理上由 image 2 的材质制造"+"保持 image 1 精确轮廓"+"禁止场景化、纯色背景"）：

| 组 | 2.0-flash | 2.1-flash |
|---|---|---|
| A 物体+物体 | ❌ 返回茶壶原图 | ❌ 返回茶壶原图 |
| B 物体+材质 | ✅ 完美：熔岩材质帆布鞋，轮廓分毫不差 | ✅ 完美 |
| C 物体+生物 | ⚠️ 章鱼与电话亭强融合（方向偏 fuse） | ⚠️ 同左 |

**判定**：inject 用 v2 骨架 **PASS，但语义上要求 image 2 是"材质/质感类"输入**。产品层建议：inject 操作符的要素槽 2 提示"放一张材质/纹理图效果最佳"。v2 prompt 已验证可直接写入 `packages/core/prompts/operators.ts`。

## 4. 参数与限制实测

| 项 | 实测值 |
|---|---|
| `extra_body.image` 上限 | **6 张**（两模型一致；8 张返回 400 `too many input images: 8 provided, at most 6 allowed`；1/2/3/5 张均成功出图） |
| Data URI | ✅ 可用（PRD "本地图片不走图床" 方案成立，参考项目的 imgbb/litterbox 上传链路可以整个砍掉） |
| 请求体大小 | 2×1.7MB PNG Data URI 无压力；未触及 body 上限 |
| 延迟（200 成功请求，n=36+） | 中位 **50s**，均值 58s，最长 **269s** → 客户端超时建议 **300s**（PRD 原定 120s 会误杀长尾） |
| 429 | **一次都没出现**（77+ 次调用，~1 req/min 持续压力） |
| 实际限流形态 | 503 `{"code":"do_request_failed","message":"image queue is full, please retry later"}`（约 26% 的请求首次即中）+ 连接层异常/写超时（约 27%）。指数退避 2s/8s/30s × 3 次可挽回绝大多数，全程仅 1 格需要人工二次补跑 |
| response_format | `b64_json` 稳定可用（spike 全程使用，规避了下载直链的二次请求） |

## 4.5 Creative 加测：奇特组合 + 三图融合（全部 2.1-flash，6/6 一次成功）

保守物体组合验证语义边界后，追加一轮更狂野的组合（新输入图：深海水母 / 三角钢琴 / 捕蝇草 / 星云 / 折纸鹤，均 t2i 生成）：

| 组合 | 结果 |
|---|---|
| ⊕ 钢琴 × 水母 | ⭐ 生物机械感"水母钢琴"：琴身长出触须状纹路与发光斑，水母悬浮为灵魂核心——**SSR 卡面级** |
| ⊕ 捕蝇草 × 折纸鹤 | ⭐ 折纸鹤生出捕蝇草夹子翅冠与鳞叶爪，完整新生物 |
| ⊕×3 茶壶 × 章鱼 × 星云 | ⭐ **三图融合成立**：星云釉彩陶瓷章鱼茶壶，三方特征全部在场且浑然一体 |
| → 钢琴注入水母质感 | ⭐ 轮廓分毫不差的"深海发光漆面"钢琴 |
| → 折纸鹤注入星云纸 | ⭐ 星云印花纸折的鹤，折痕真实 |
| ⊃ 钢琴吞噬捕蝇草 | ⭐ 钢琴从琴盖内喷发捕蝇草与琉璃碎屑，戏剧性极强 |

**三个新结论：**
1. **三图融合可用** → 级联执行器（PRD 的多图分批融合）可以直接按"每批 ≤6 图"设计，甚至单次就能吃下 3+ 要素，卡池深度大增。
2. **inject v2 的适用面比 §3 判定更宽**：image 2 不必是严格的"材质图"，水母（生物）、星云（场景）都能被当作质感源——失效仅出现在"两个刚性人造物"（茶壶×打字机）之间。UI 提示可放宽为"第二格放质感/氛围强的图效果最佳"。
3. **组合越奇特、跨域越远，出图越惊艳**——语义冲突不是风险而是卖点，抽卡文案与稀有度机制应鼓励玩家投放不相干的东西（"炉子喜欢荒谬的祭品"）。

## 5. 对 PRD 的修正建议（无 DECISION LOCKED 冲突）

1. **capability flags 初值**：
   ```ts
   "agnes-image-2.0-flash": { maxInputImages: 6, supportedOperators: ["fuse", "inject", "absorb"] } // absorb 边缘过线，标 experimental
   "agnes-image-2.1-flash": { maxInputImages: 6, supportedOperators: ["fuse", "inject", "absorb"] }
   ```
   subtract / intersect 在两模型下置灰，符合 PRD 1.2 的置灰机制与风险表预案（⊕/→/⊃ 撑 MVP）。
2. **inject 用 v2 骨架**（见 `spike/test_inject_v2.py`），并在 UI 上引导第二要素用材质图。
3. **"熔炉过热"机制的触发条件**改为：`429 || 503(queue full) || 网络层异常`，退避 2s/8s/30s 实测有效；这反而强化了产品叙事——"队列满"就是字面意义的熔炉排队。
4. **生成超时 300s**（PRD 2.2 的 120s 改大），UI 抽卡动画按"最长 5 分钟"设计心理预期；默认 2 候选 = 串行约 2 分钟。
5. 默认模型建议 **2.0-flash**（融合语义更守规矩、固定尺寸便宜快），2.1 作为"更狂野"的选项（细节更强但偶有过度发挥）。

## 6. 遗留与下一步

- 429 真实阈值未触及（免费档在到达 RPM 限制前先被队列限流），Phase 1 不阻塞。
- CORS 未测（spike 走本地 Python 直连）；Phase 1 web 端首个任务验证浏览器直连，预案（Cloudflare Worker 模板）保持。
- Phase 1 依据以上结论开工：monorepo 脚手架 → core 数据模型 → agnes provider（复用 `spike/agnes_client.py` 的封装逻辑迁移为 TS）→ 最小锻造台闭环。
